# Smithers PRD v2 — YOLO Agent Terminal Workbench

> Generated by design agent. See feedback section for v1 learnings.

---

## 0) Feedback on v1 doc (what I'd change)

1. **Approvals/permissions were the wrong center of gravity.**
   You want **YOLO** and believe approvals are not real security. Design should treat "security" as **sandbox boundaries + minimal host exposure**, not per-tool prompts.

2. **I over-indexed on slash commands.**
   Your direction is "skills only." The UI/UX should make "skills" discoverable, composable, and automatable—without requiring users to memorize syntax.

3. **I didn't lean hard enough into "this is a terminal product."**
   "Open the sandbox terminal" should be **one click from anywhere** (header, tool card, message node, subagent node).

4. **JJ needs to be the first-class substrate for history/checkpoints/stack behavior**, and it should be **invisible unless the user asks**.
   Users should see "checkpoints," "stack," "rebase," "sync," "restore," not "jj op log" or "jj rebase." JJ becomes the implementation detail.

5. **Tree-structured sessions + stacked commits need a dedicated visualization.**
   This can't be an afterthought: you want fork visualization, prompt rebasing, and subagent graphs. That demands a **Graph view** that's as primary as Chat view.

6. **Sandboxing choice (bubblewrap) has a macOS reality constraint.**
   bubblewrap is a Linux sandboxing tool and does not support macOS. ([GitHub][1])
   So the engineering spec must explicitly describe how to deliver bwrap semantics on macOS (e.g., Linux VM/container layer).

# 1) PRD v2 — Smithers (YOLO agent terminal workbench)

## 1.1 Product definition

**Smithers** is a native macOS app that runs coding agents against a repo inside **sandboxed workspaces**, presenting the work as:

* a **high-quality chat transcript** (markdown-first),
* a **terminal-first workbench** (one-click drill into sandbox terminals),
* a **graph/stack UI** for forks, rebases, checkpoints, and subagents.

MVP uses the Claude Agent SDK for speed, but Smithers is built around an **internal Agent Runtime Protocol** so your in-house Python agent can replace Claude SDK without UI churn.

## 1.2 Non-goals (explicit)

* No per-tool "Allow/Deny" prompts in MVP (**YOLO only**).
* No user-facing "git commit management" UI; commits/snapshots are **backgrounded** and represented as **checkpoints / stack nodes**.
* No slash commands. Skills only.

## 1.3 Pillars

### Pillar A — YOLO + real isolation

Security posture is **sandboxing + constrained mounts**, not user approvals. bubblewrap is a "tool for constructing sandbox environments," not a complete policy by itself; Smithers defines the policy. ([GitHub][2])

### Pillar B — Terminal-native ergonomics

Chat is the narrative, terminal is the control surface.

* Any sandbox (main agent, subagent, historical node) is **one click away** from an interactive terminal.

### Pillar C — JJ as invisible time machine

JJ underpins:

* checkpoints (automatic snapshots),
* stack operations (rebase/squash/fixup),
* prompt rebasing (rerun from node with improved prompts).

JJ has strong built-in "operation log snapshots" and automatic working-copy snapshotting concepts we can leverage. ([GitHub][3])
But users see "Checkpoint/Restore," not "op log."

### Pillar D — Graph-first visibility

The "truth" of the session is a **graph**:

* conversation branches,
* code state nodes (stack),
* subagent task graph.

Chat is a projection of that graph.

### Pillar E — Skills as the only "commands"

Skills are:

* UI-invoked workflows (e.g., "Summarize," "Plan," "Prompt Rebase," "Create Form," "Generate Diff Summary," "Run Tests," "Prepare Handoff"),
* agent-invoked capabilities (later, when you build your own orchestrator).

### Pillar F — Cloud sharing + local-first backups

Local event log + artifacts are canonical; cloud sync mirrors them for:

* "thread sharing" links,
* handoffs between machines/people,
* CRDT-like convergence (practically: DAG replication + deterministic merge).

## 1.4 Core concepts

### Workspace

A workspace = (Repo root) + (Sandbox policy) + (JJ-backed history) + (Sync config).
Workspaces are where code changes happen.

### Sandbox

A sandbox is a constrained execution environment for tools:

* file mounts (workspace RW, everything else RO or hidden),
* network policy (toggle),
* process containment.

### Session graph

A session is a **DAG** of nodes:

* user/assistant messages,
* tool invocations/results,
* checkpoints (code snapshots),
* subagent runs,
* prompt rebase "replays."

### Stack (Graphite-inspired)

A "stack" is an ordered set of checkpoints/changes:

* rebasing "everything above" is a first-class operation,
* auto-rebase can be triggered by remote changes,
* AI assistance can propose conflict resolutions.

### Surfaces (context you can attach)

Surfaces are attachable context sources:

* current workspace diff
* a tool output viewer
* a browser tab snapshot
* a terminal transcript snapshot
* other sessions (explicitly attached, not implicit)

## 1.5 MVP scope (the thing you ship soon)

### Must-have

1. **Chat transcript with streaming + tool timeline**

   * Markdown rendering: extremely high quality (code blocks, tables, lists, links, images later)
   * Tool calls and results as structured, collapsible cards
   * Output viewer for large payloads

2. **Sandbox-first tool execution**

   * Run Read/Edit/Bash/Glob/Grep/WebFetch/etc inside sandbox
   * One-click open terminal into the sandbox from:

     * session header
     * each tool card
     * each graph node
     * each subagent run

3. **JJ-based checkpoints**

   * Automatic checkpoints (policy-driven; e.g., before/after edit batches)
   * "Restore to checkpoint" with clear UI

4. **Graph view (trees + stack)**

   * View forks/branches of the conversation
   * View the code stack (ordered checkpoints)
   * Jump from any node to chat position + code state

5. **Todos panel**

   * User/agent can create items
   * Attach todos to context ("agent, follow these")

6. **Skills system**

   * Built-in skills: Summarize, Plan, Create Form, Prompt Rebase (v0), Generate Diff Summary, Prepare Handoff
   * No text-based slash interface—pure UI affordances + keyboard palette

7. **Search**

   * Global search across sessions/messages/tool outputs/diffs (fast, incremental)

8. **Session naming**

   * Auto-title using cheap/fast model unless user names it

9. **AGENTS.md**

   * Auto-detected and injected as instruction context (no Claude.md requirement)

### Nice-to-have (still "near MVP")

* MCP server configuration UI per workspace/session (simple)
* Built-in browser surface (basic)
* Thread sharing (export + cloud sync skeleton)

## 1.6 Key product decisions (updated answers)

1. **Very long tool outputs**
   Always truncate inline, store full content as an artifact, open in **Output Viewer** with search/jump/copy. No "infinite scroll inside chat."

2. **Tool calls inline vs separate panel**
   Inline in transcript for chronology + optional Inspector for full details. Tool output viewer is separate.

3. **Cost/token usage**
   Not a primary UI element in MVP. Put it in an Inspector "Run Details" view if available; otherwise omit.

4. **Streaming markdown**
   Stream as plain text + minimal formatting; finalize render after completion. Claude Agent SDK supports partial streaming via `include_partial_messages`. ([Claude][4])

5. **Errors**
   Errors are transcript events + sticky run banner. Provide "Retry from node" and "Fork from node." No permission prompts; failures are either sandbox/tool/runtime.

6. **Auto-save**
   Always. Sessions are append-only event logs; graph is durable. "Save" doesn't exist; "Export/Share" exists.

# 2) Design Spec v2 — UX/UI (chat + terminal + graph + skills)

## 2.1 Layout: "terminal workbench" not "chat app"

### Primary window: 3-column + terminal drawer

* **Left sidebar (260–320pt):** Workspaces + Sessions + Search
* **Center:** Chat or Graph (toggle)
* **Right Inspector (320–420pt):** Tabs: Stack, Diff, Todos, Tools, Browser, Subagents, Run Details
* **Bottom drawer:** Terminal (dockable), defaults collapsed

Key: the user can keep chat front-and-center but can summon terminal/stack/diff instantly.

## 2.2 Navigation model

### Top-level modes

1. **Chat Mode** (default): chronological transcript
2. **Graph Mode**: DAG view of session + stack + subagents

Both modes share the same selection state: selecting a node updates inspector, terminal, diff, etc.

### Selection is everything

Selection can be:

* a message node
* a tool invocation node
* a checkpoint node
* a subagent run node
* a browser snapshot node

Everything else (Inspector tabs, terminal directory, diff base) follows selection.

## 2.3 Component hierarchy (SwiftUI)

### Root

* `SmithersRootView`

  * `WorkspaceSidebarView`
  * `MainPaneView` (Chat/Graph)
  * `InspectorPaneView`
  * `TerminalDrawerView`

### Sidebar

* `WorkspaceSidebarView`

  * `WorkspacePicker` (workspaces)
  * `SessionSearchField`
  * `SessionList`

    * grouped + pinned + recents
  * `SidebarFooter` (Settings, MCP, Cloud)

### Main pane

* `MainPaneView`

  * `SessionHeaderBar`

    * Title (editable)
    * Run state indicator (Idle/Running/Stopped/Error)
    * Buttons:

      * **Terminal** (opens drawer into current sandbox)
      * **Graph/Chat toggle**
      * **Share**
      * **Fork**
  * `ChatPaneView` OR `GraphPaneView`
  * `ComposerBarView`

### Chat pane

* `ChatPaneView`

  * `VirtualizedMessageList`
  * `JumpToLatestButton` (when scrolled up)

### Graph pane

* `GraphPaneView`

  * `SessionGraphCanvas`

    * lanes: Main thread, Branches, Subagents
    * overlays: checkpoints, stack nodes, prompt rebase nodes
  * `MiniMap`
  * `FilterBar` (hide tool nodes, show only checkpoints, etc.)

### Inspector

* `InspectorPaneView`

  * Tabs:

    * **Stack**
    * **Diff**
    * **Todos**
    * **Tool Details**
    * **Browser**
    * **Subagents**
    * **Search Results**
    * **Run Details / Logs**

### Terminal drawer

* `TerminalDrawerView`

  * `SandboxTerminalTabs`

    * "Main"
    * "Subagent X"
    * "Checkpoint N (read-only)" (optional)
  * Each tab is a terminal session attached to a sandbox or snapshot

## 2.4 Message rendering specs (high quality)

### User message

* Right aligned bubble
* Simple markdown-lite (inline code, links)
* Timestamp on hover

### Assistant message

* Left aligned; max width ~80%
* **Markdown renderer** supports:

  * headings, lists, tables, blockquotes
  * fenced code blocks with syntax highlighting
  * inline code
* Per-block affordances:

  * copy code
  * open as file (if detected)
  * "Pin as artifact" (saves to artifacts list)

### Tool cards (inline, collapsible)

Tool card header:

* icon + tool name
* short summary (e.g., `Bash: jj status` / `Read: src/auth.py`)
* status (running/success/error)
* actions:

  * **Open Terminal Here**
  * Copy input
  * Open details in Inspector

Expanded view:

* Input params (formatted JSON)
* Output preview (truncated)
* "Open Output Viewer" if large
* For file edits: show mini diff snippet + "Open Diff tab"

### Large outputs

Inline preview shows:

* first N lines
* "+ 18,240 more lines"
* open in Output Viewer (search, line numbers, copy, save)

### Thinking/reasoning

Default: not shown in chat unless you choose to expose it.
If exposed: collapsed "Thought" disclosure that renders in muted color.

## 2.5 Terminal drill-in (core terminal feel)

**Everywhere** you see an execution context, you see a Terminal button:

* Session header: opens terminal in the current workspace sandbox root
* Tool card: opens terminal at tool working directory (or with the command prefilled)
* Graph node: opens terminal pinned to that node's sandbox
* Subagent node: opens terminal inside that subagent sandbox

Terminal drawer behavior:

* opens at ~35% height; resizable; can "pop out" into a new window/tab

## 2.6 JJ-based "Checkpoints" UX (JJ hidden)

### Checkpoints in UI

* Shown as "Checkpoint markers" on the transcript and as nodes in Graph view.
* Each checkpoint has:

  * short auto-generated label ("After tests fixed", "Before refactor")
  * timestamp
  * "Restore" action
  * "Fork from here" action

JJ capabilities we rely on (implementation detail):

* JJ records snapshots tied to operations and working copy changes. ([GitHub][3])

### Stack view (Graphite-inspired)

Inspector → **Stack** shows a vertical list:

* Node 1: Base
* Node 2: "Fix auth bug"
* Node 3: "Refactor sessions"
* Node 4: "Add MCP UI"

Actions (UI words, not git words):

* Rebase stack onto latest
* Squash nodes
* Split node
* Move node up/down
* "AI help: resolve conflicts / rewrite descriptions"

## 2.7 Fork visualization (trees)

Forking happens from:

* any user message
* any assistant result
* any checkpoint

Graph view makes forks explicit:

* branch lanes
* clear "active branch" highlight
* "compare branches" in Diff tab

## 2.8 Skills (no slash commands)

### Skills palette

* Open via toolbar button or `⌘K`
* Searchable list of skills
* Skill execution modes:

  1. **Side action** (does not go into chat unless you "Insert result")
  2. **Agent run** (creates a new run node; can optionally append to chat)

### Built-in MVP skills

* Summarize session
* Generate handoff (shareable)
* Plan (asks clarifying questions)
* Create Form
* Prompt Rebase
* Diff Summary
* Rename session

### Skill results are artifacts

Skill results land in:

* Inspector "Artifacts" section
* can be inserted into chat as context

## 2.9 Prompt rebasing (first-class)

UI:

* From any node: "Prompt Rebase"
* Creates a new branch that:

  * replays the conversation up to that node
  * injects hindsight learnings as system guidance
  * reruns the agent
* Graph view shows "Rebase Replay" nodes and their deltas
* Diff tab compares "original run" vs "rebased run" changes

## 2.10 Create Form tool (plan mode)

When agent calls Create Form:

* UI renders a small HTML form in Inspector → Browser/Forms tab
* User fills it
* Submission produces structured data (JSON) returned to agent
* Form is stored as an artifact

Design constraints:

* forms are sandboxed in a webview
* only local HTML/JS allowed (no remote scripts)
* strong visual consistency with macOS controls

## 2.11 Built-in browser ("Chrome") surface

Pragmatic design:

* Built-in browser panel with tabs
* "Send page to agent" captures:

  * URL
  * readability/extracted text
  * optional screenshot snapshot artifact
* "Open in system browser" as escape hatch

This gives you the "navigate somewhere then tell agent to look at it" workflow without needing external Safari integration as a dependency for MVP.

## 2.12 Search (must be excellent)

Single global search box that returns:

* sessions by title
* messages
* tool invocations
* tool outputs (indexed previews + on-demand full file search)
* checkpoint labels
* diffs

Search results are navigable as graph nodes.

## 2.13 Subagents (future agent model compatibility)

UI support now:

* show "subagent run cards" in transcript (collapsed)
* Graph mode shows subagent nodes and edges
* Drill-in opens:

  * subagent transcript
  * subagent terminal
  * subagent artifacts

This sets you up for your Python agent's "graphs of agents" without redesign.

# 3) Engineering Spec v2 — Architecture, sandboxing, JJ, protocol, performance

## 3.1 Foundational decision: internal runtime protocol

Even if MVP uses Claude Agent SDK, Smithers should implement:

* `AgentEngine` interface in Swift
* `agentd` (Python) that speaks Smithers protocol
* adapters:

  * Claude SDK adapter (now)
  * Smithers custom agent adapter (later)

This prevents "Claude SDK message types" from leaking into UI.

## 3.2 Sandboxing with bubblewrap on macOS (reality + plan)

### Reality

bubblewrap does **not** support macOS. ([GitHub][1])
Therefore, to use bubblewrap for real isolation, you need a **Linux execution environment** on macOS.

### Proposed MVP architecture (recommended)

* Ship a minimal **Linux VM** managed by Smithers (Apple Virtualization Framework).
* Inside the VM:

  * `sandboxd` (a small daemon)
  * bubblewrap available
  * jj available
  * optional: headless tooling (ripgrep, fd, node/python toolchains)

Smithers on macOS:

* mounts workspace directories into VM (virtiofs)
* requests sandbox creation via a control channel (vsock or unix socket)
* gets back a `sandbox_id` + attach endpoints (pty/ssh/vsock)
* all tool execution happens inside bwrap within VM

Why this is compatible with YOLO:

* the user trusts Smithers; isolation protects host by construction, not by prompts.

### Sandbox policy (what we enforce)

bubblewrap is a constructor; Smithers defines the policy. ([GitHub][2])
Baseline policy per sandbox:

* RW mount: workspace path only
* RO mounts: toolchain dirs as needed
* No mount: host home, Documents, SSH keys, etc.
* Network: configurable (default ON for dev ergonomics; OFF option for tighter isolation)
* Resource limits: cpu/mem/timeouts per tool
* Dedicated tmp/home per sandbox

## 3.3 One-click terminal into any sandbox

Mechanically:

* `sandboxd` exposes a PTY attach method:

  * `sandbox.attach_pty(sandbox_id, cwd, env)`
* SwiftUI TerminalDrawer creates a terminal session connected to that PTY.
* "Open Terminal Here" simply sets `cwd` (tool working directory or node root) and opens drawer.

This also supports "terminal per subagent sandbox."

## 3.4 JJ integration (checkpoints + stack + autosync)

### JJ as the storage primitive for code state

We treat JJ as:

* checkpoint engine
* stack engine (Graphite-like)
* rebase engine
* sync engine

JJ provides an operation log where each operation contains a snapshot ("view") of repo state. ([GitHub][3])
JJ also snapshots the working copy as part of normal usage patterns. ([JJ VCS Documentation][5])

### How Smithers uses JJ without exposing JJ

We create a `RepoStateController` that offers high-level operations:

* `createCheckpoint(label, reason, nodeId)`
* `restoreCheckpoint(checkpointId)`
* `createStackNode(label, fromCheckpoint)`
* `rebaseStack(onto)`
* `squashNodes(nodeIds)`
* `splitNode(nodeId, strategy)`
* `syncPush()` / `syncPullRebase()`

Under the hood this maps to JJ commands and revsets, but the UI never shows them.

### Graphite-inspired auto-rebase

Graphite popularizes stack maintenance via auto-rebase of "upstack" branches after merges. ([Graphite][6])
Smithers will implement the analogous concept:

* on remote update:

  * update base
  * rebase stack nodes "above" base
  * if conflicts: pause + ask agent to resolve (YOLO still, but with visibility)

### Auto-push / auto-pull

You want "every time we commit/snapshot we push" and "when others push we auto-rebase."
Implement as a background `SyncDaemon` per workspace:

* push on new checkpoint nodes (debounced)
* poll or webhook-like fetch (poll for MVP)
* apply rebase automatically
* emit transcript/system events: "Synced • Rebasing stack (3 nodes)"

## 3.5 Session graph + event sourcing (durable, cloud-friendly)

### Data model (core)

* `SessionGraph` is persisted as an **append-only event log** (SQLite).
* Node types:

  * MessageNode(user/assistant)
  * ToolNode(tool_use/tool_result)
  * CheckpointNode
  * SubagentNode
  * SkillRunNode
  * PromptRebaseNode
  * BrowserSnapshotNode
  * TerminalSnapshotNode

### Why event log

* crash-safe
* natural for streaming
* natural for cloud replication (send missing events)

### Cloud sync / CRDT-ish behavior

Treat the session graph as a DAG of immutable nodes:

* merge = union of nodes + deterministic conflict resolution for mutable metadata (titles, pins)
* replication can be "send events since cursor"
* artifacts referenced by content-addressed keys

This enables "thread sharing" as a stable object:

* share link points to a session graph + artifacts, plus optional repo pointers (workspace id, stack node id)

## 3.6 Agent Runtime Protocol (Swift ↔ Python)

### Transport

Keep it simple and robust:

* Python subprocess on macOS (for now) **or** inside VM sidecar
* NDJSON over pipes or socket
* Requests are JSON-RPC-ish; events are notifications

### Message families

1. **Run control**

* `run.start(session_id, node_parent, input, attached_surfaces, skill?)`
* `run.cancel(run_id)`
* `run.pause` (optional later)

2. **Streaming**
   Claude Agent SDK supports partial streaming via `include_partial_messages`. ([Claude][4])
   We represent streaming as:

* `assistant.delta(text)`
* `assistant.final(message_id)`

3. **Tools**

* `tool.start(tool_use_id, name, input, sandbox_id)`
* `tool.output_ref(tool_use_id, artifact_ref, preview)`
* `tool.end(tool_use_id, status)`

4. **Checkpoints/stack**

* `checkpoint.created(checkpoint_id, label, stack_position)`
* `stack.rebased(details)`
* `sync.status(state)`

5. **Subagents**

* `subagent.start(subagent_id, parent_run_id, label)`
* `subagent.end(subagent_id, status)`
* `subagent.graph(edge_events…)`

6. **Skills**

* `skill.start(skill_id, mode, params)`
* `skill.result(artifact_ref, summary)`
* `skill.end`

7. **Create Form**

* `form.create(form_id, html, schema, submit_target)`
* `form.submit(form_id, data_json)`

## 3.7 Tool output storage (never inline big blobs)

* Tool output is always stored as an artifact file:

  * content-addressed key (hash)
  * preview stored in DB
  * full file stored on disk (and optionally in cloud)

Output Viewer loads the artifact on demand and supports:

* incremental search
* jump to line
* copy selection

## 3.8 High-end markdown rendering (performance + fidelity)

### Requirements you set

* "Extremely high quality markdown viewing"
* Works for long sessions
* Doesn't lag while streaming

### Proposed rendering pipeline

1. During streaming: render plain text (fast append).
2. On finalize:

   * parse markdown to AST
   * render to attributed runs
   * apply:

     * typography rules
     * code block styling
     * syntax highlighting (tree-sitter/highlighter)
   * cache rendered layout keyed by message hash

### Virtualized list

SwiftUI `LazyVStack` is usually fine if cells are lightweight.
But "high fidelity markdown" can make cells heavy. Plan:

* **MVP:** SwiftUI LazyVStack + cached attributed strings + "open in viewer" for huge messages
* **If needed:** drop in an AppKit-backed `NSCollectionView` for true view reuse/virtualization (still hosted inside SwiftUI). This is one of the rare "UIKit/AppKit is justified" cases for performance.

## 3.9 Built-in browser surface implementation

* Use a webview surface with:

  * capture URL
  * extract readable text (DOM extraction)
  * optional screenshot snapshot
* Store snapshots as artifacts; attach to agent as surfaces.

This is also the foundation for "agent looks at browser."

## 3.10 Search (excellent, global, fast)

Storage:

* SQLite + FTS for:

  * message text
  * tool input params (stringified)
  * tool output previews
  * checkpoint labels
  * skill results summaries

For large artifacts:

* index previews + optionally background index full text into chunk table.

UI:

* search results are graph nodes; selecting result jumps chat/graph and updates inspector.

## 3.11 MCP server configuration UI

Agent SDK supports MCP as a way to connect tools. ([Claude][7])
Even if your future agent differs, you want the *concept*:

* per-workspace MCP servers
* per-session enablement
* GUI configuration, no manual JSON required (though you can still import/export)

Implement an internal "Tool Registry" abstraction that can represent:

* built-in tools
* MCP tools
* Smithers-native tools (Create Form, ReadOtherChat, CaptureBrowser, etc.)

## 3.12 "Side actions" (system operations not in chat)

Implement as **Operations**:

* run in background
* produce artifacts
* optionally "Insert into chat" as a message or context attachment

Examples:

* Summarize chat
* Prepare handoff
* Generate PR description
* Analyze stack health
* Cost breakdown (if you ever want it)

## 3.13 Prompt rebasing engineering

Prompt rebasing is implemented as:

* create a new branch node
* replay selected portion of graph into a new run context
* attach hindsight artifact(s) as system guidance
* run agent from scratch in new sandbox state (or restored checkpoint)

Everything is visible as graph nodes; nothing is "magic."

## What I would do next (concretely)

If you want the team to execute quickly with minimal churn:

1. **Lock the core abstractions now**

* `SessionGraph` + event store
* `ArtifactStore`
* `SandboxRuntime` interface
* `AgentRuntimeProtocol`
* `RepoStateController` (JJ-backed)

2. **Build the "one-click terminal" loop immediately**
   It is the differentiator that makes this feel like a terminal product instead of another chat client.

3. **Ship Graph Mode early**
   Don't wait. Trees/forks/subagents/stacks/prompt rebases all become cheap once graph selection is first-class.

4. **Implement checkpoints as JJ-backed from day one**
   It will prevent you from designing UI around git semantics you don't want.
