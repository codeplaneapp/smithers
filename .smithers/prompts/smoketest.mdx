npm package. Your job is to prove that a user who has never touched
Smithers can install it and exercise the feature set — and that every
claim in the changelog below actually holds against the published
`smithers-orchestrator@{props.version}` on npm.

## Setup — always work in a brand-new folder

1. Before doing anything else, clear caches so you're testing the
   registry, not a stale copy:
   - `bun pm cache rm`
   - macOS: `rm -rf $(echo $TMPDIR)bunx-*smithers-orchestrator*`
     (or check `/var/folders/**/T/bunx-*smithers-orchestrator*`).
   - Linux / other: `rm -rf $TMPDIR/bunx-*smithers-orchestrator*`.
2. Create a fresh scratch directory outside any existing project:
   `SMOKE_DIR=$(mktemp -d)/smoketest && mkdir -p "$SMOKE_DIR" && cd
   "$SMOKE_DIR"`. Never reuse an existing folder. Never work in the
   monorepo itself.
3. Run `bunx smithers-orchestrator@latest init` from the scratch
   directory. Capture stdout/stderr.

## Baseline tests — run these every time

Do each, record pass/fail with evidence (exact command, exit code,
relevant output lines):

- `bunx smithers-orchestrator@latest init` succeeds on a fresh folder.
  Confirm `.smithers/package.json` exists, `.smithers/workflows/`
  contains workflow files, and the `install` result reports
  `status: ok` (i.e. `bun install` ran inside `.smithers/`).
- `.smithers/node_modules/skills/` exists with at least one
  `<name>/SKILL.md` inside — proving the `github:mattpocock/skills`
  dep was fetched (runtime auto-install can't do that).
- Both `$(echo $TMPDIR)bunx-*smithers-orchestrator*/node_modules/.bin/smithers`
  AND `.smithers/node_modules/.bin/smithers` symlink to
  `smithers-orchestrator/src/bin/smithers.js` (not
  `@smithers-orchestrator/cli/src/index.js`). Use `readlink`.
- `bunx smithers-orchestrator@latest workflow grill-me` loads without
  crashing. The agent call itself is allowed to fail for unrelated
  reasons (API auth, scorer schema errors). What matters is that you
  see `workflowName=grill-me` and `→ grill-me:grill (attempt 1,
  iteration 0)` in the output, and that you do NOT see any of:
  - `Cannot find module 'skills/...'`
  - `TypeError: resolveEventTimeStamp is not a function`
  - `TypeError: resolveEventType is not a function`
  - `Invalid hook call. Hooks can only be called inside of the body of
    a function component.`
  - `dispatcher.useContext is not a function`
  Any of those is a **regression** and the smoke test fails.

## Changelog-driven tests — one per bullet above

For every bullet under `## Fixes` or `## Features` in the changelog
embedded above, construct a minimal reproduction of the *original* bug
and verify it no longer reproduces on
`smithers-orchestrator@{props.version}`. Common patterns:

- Fix says a dependency was missing from a generated file → open the
  file after `init` and grep for the dep.
- Fix says a command crashed → run that exact command and confirm no
  crash.
- Feature introduces a new command or flag → run it with `--help` and
  exercise at least one non-trivial path.

Be ruthless: a changelog bullet that claims "X now works" but X still
fails is a smoke test failure, even if every baseline test passed.

## Output

Return a single JSON object matching the required output schema.

- `passed`: `true` only if every baseline test AND every
  changelog-derived test passed. Any regression → `false`.
- `summary`: 1-3 sentences. What worked, what broke.
- `findings`: one entry per test. `area` is a short label
  (e.g. `init.install`, `workflow.grill-me`,
  `changelog.{props.version}.bin-delegation`). `status` is
  `pass | fail | skipped`. `evidence` is the exact command + the key
  line of output that proves the result. Never skip a changelog
  bullet; if a bullet is genuinely impossible to reproduce as a user
  (e.g. internal CI-only notes), still emit a finding explaining why.
- `reproSteps`: if `passed` is `false`, a numbered list of shell
  commands someone can paste to reproduce each failure from a clean
  machine.
