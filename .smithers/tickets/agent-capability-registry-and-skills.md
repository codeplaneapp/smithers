# First-class agent capability registry and Smithers skills

## Problem

Smithers has the pieces for agent-specific tooling, but they are fragmented:

- `AgentLike.tools` is the only generic capability field the engine knows about.
- `BaseCliAgent` hardcodes `readonly tools: Record<string, never> = {}`.
- The engine uses `agent.tools` for cache signatures, so all CLI agents collapse
  to the same empty tool fingerprint even when their actual tool surfaces differ.
- Agent-specific features are scattered across adapter flags:
  - Claude: MCP config files, allowed/disallowed tools
  - Codex: config overrides, sandbox flags, JSONL event stream
  - Gemini: allowed MCP server names, built-in tools
  - Kimi: MCP config files and `skillsDir`
  - PI: built-in tools, extensions, skills, prompt templates, UI requests
- Docs already talk about skills and integration sync, but there is no canonical
  Smithers skill-pack story and no `skills add` command implementation.

This makes capability-aware routing, cache invalidation, docs generation, and
agent selection much weaker than they should be.

## Proposal

Introduce a first-class capability registry for every agent adapter, then ship a
real Smithers skills distribution path on top of it.

### Capability model

Add a capability shape that is richer than `tools`:

```ts
type AgentCapabilityRegistry = {
  version: 1;
  engine: "claude-code" | "codex" | "gemini" | "kimi" | "pi" | "amp";
  runtimeTools: Record<string, AgentToolDescriptor>;
  mcp: {
    bootstrap: "inline-config" | "project-config" | "allow-list" | "unsupported";
    supportsProjectScope: boolean;
    supportsUserScope: boolean;
  };
  skills: {
    supportsSkills: boolean;
    installMode?: "files" | "dir" | "plugin";
    smithersSkillIds: string[];
  };
  humanInteraction: {
    supportsUiRequests: boolean;
    methods: string[];
  };
  builtIns: string[];
};
```

The registry should be exposed on agent instances and consumed by:

- engine caching and task metadata
- `smithers ask`
- agent diagnostics
- skills installation
- docs and prompt generation

### Engine changes

Replace the current `Object.keys(agent.tools)` signature with a capability-aware
fingerprint:

```ts
const toolSig = hashCapabilityRegistry(agent.capabilities ?? null);
```

This lets cache keys change when:

- the available Smithers MCP server set changes
- a skill pack is installed or removed
- human UI support differs by adapter
- a built-in tool allow-list changes

### Smithers skill packs

Ship real Smithers-owned skill packs for agents that support them.

The skills should teach behavior, not implement actions:

- workflow-first planning
- when to run a workflow vs edit directly
- how approvals and human tools work
- how to prefer semantic Smithers tools over raw CLI tools
- how to interpret run, node, approval, and artifact state

Do not encode durable operations only in prompt text. Skills should pair with
real tools from the semantic MCP layer.

### New CLI features

Implement the docs-promised and capability-aware command family:

- `smithers skills add [--agent ...] [--scope ...]`
- `smithers skills doctor`
- `smithers agents capabilities`
- `smithers agents doctor`

These commands should be generated from the actual adapter capability registry,
not from hardcoded per-agent assumptions scattered around the CLI.

## Rollout phases

### Phase 1: Registry model

- Extend `AgentLike` with `capabilities?: AgentCapabilityRegistry`.
- Update every built-in CLI agent adapter to populate the registry.
- Update engine cache signatures and metadata to use the registry.

### Phase 2: Diagnostics and routing

- Add capability-aware agent diagnostics commands.
- Make `detectAvailableAgents()` emit capability bootstrap hints.
- Show the capability registry in `workflow doctor` and `ask --list-agents`.

### Phase 3: Skill-pack installation

- Implement `smithers skills add`.
- Ship Smithers skill-pack templates per supported agent ecosystem.
- Add `smithers skills doctor` to explain what is installed and what is missing.

## Additional steps

1. Preserve `agent.tools` for AI SDK compatibility, but stop treating it as the
   only capability signal.
2. Add a shared `hashCapabilityRegistry()` helper.
3. Update task metadata to record the active capability registry fingerprint.
4. Surface capability mismatches in `workflow doctor`.
5. Reconcile docs examples that reference ghost plugins or skills that are not
   generated by Smithers itself.
6. Add a generated `skills/` directory in `.smithers/` only if the install
   command needs workspace-local assets.
7. Keep skill content small and versioned so agents can reason about what Smithers
   expects without duplicating the whole docs site.

## Verification requirements

### Unit and integration tests

1. **Capability fingerprint changes when MCP wiring changes** - Two agents with
   different MCP bootstrap modes produce different fingerprints.
2. **Capability fingerprint changes when skills differ** - Adding or removing a
   Smithers skill changes the fingerprint.
3. **CLI agents expose registries** - Claude, Codex, Gemini, Kimi, and PI all
   populate `capabilities`.
4. **Engine cache uses registry fingerprint** - Cached task metadata changes when
   the capability registry changes, even if `agent.tools` stays empty.
5. **`agents capabilities` output** - Command prints a stable JSON report of each
   installed adapter's MCP, skills, and UI support.
6. **`skills add` installs only supported artifacts** - Unsupported agents are
   reported cleanly; supported agents receive the right files/config.
7. **`skills doctor` detects drift** - Missing or stale installed skill files are
   reported.

### Corner cases

8. **Mixed support matrix** - Some agents support MCP but not skills; others
   support skills but not project-scoped config. Output stays correct.
9. **No writable skill destination** - `skills add` fails clearly without partial
   installs.
10. **Backward compatibility** - AI SDK agents without registries still work and
    hash to a stable fallback fingerprint.

## Observability

### New events

- `AgentCapabilitiesResolved { agentId, engine, fingerprint, timestampMs }`
- `SkillPackInstalled { agentId, skillIds, scope, timestampMs }`
- `SkillPackDriftDetected { agentId, missingFiles, timestampMs }`

### New metrics

- `smithers.agent.capabilities_resolved_total`
- `smithers.agent.capability_fingerprint_changes_total`
- `smithers.skills.install_total`
- `smithers.skills.install_failures_total`
- `smithers.skills.drift_total`

### Logging

- `Effect.withLogSpan("agent:capabilities")`
- `Effect.withLogSpan("skills:install")`
- Annotate with `{ engine, fingerprint, scope }`

## Effect.ts architecture

The capability registry, skill installation, and cache-signature plumbing should
all remain Effect-first. Only the CLI entrypoints and test harnesses should call
`runPromise()`.
