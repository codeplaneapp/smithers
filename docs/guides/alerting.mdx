---
title: Alerting
description: Declare typed alert policy defaults in createSmithers() and override them per workflow.
---

Use `alertPolicy` when you want a workflow to carry operational alert metadata such as ownership, severity, runbooks, labels, and deterministic reactions.

The API is declarative:

- `createSmithers(..., { alertPolicy })` defines module-level defaults.
- `smithers(..., { alertPolicy })` overrides those defaults for one workflow.
- `deliver` uses logical destination names such as `oncall` or `author`; Smithers does not ship a built-in Slack or PagerDuty client.

## Module Defaults

```tsx
import {
  createSmithers,
  type SmithersAlertPolicy,
} from "smithers-orchestrator";
import { z } from "zod";

const platformAlerts: SmithersAlertPolicy = {
  defaults: {
    owner: "platform",
    severity: "warning",
    runbook: "https://internal/runbooks/smithers-workflows",
    labels: {
      service: "deploy-bot",
      env: "prod",
    },
  },
  reactions: {
    "notify-author": { kind: "deliver", destination: "author" },
    "notify-oncall": { kind: "deliver", destination: "oncall" },
  },
};

const { Workflow, Task, smithers, outputs } = createSmithers(
  {
    deployResult: z.object({
      ok: z.boolean(),
      revision: z.string(),
    }),
  },
  {
    alertPolicy: platformAlerts,
  },
);
```

## Workflow Overrides

```tsx
export default smithers(
  () => (
    <Workflow name="deploy">
      <Task id="deploy" output={outputs.deployResult} agent={deployAgent}>
        Deploy the release and report the final revision.
      </Task>
    </Workflow>
  ),
  {
    alertPolicy: {
      rules: {
        runFailed: {
          severity: "critical",
          reaction: "notify-oncall",
        },
        approvalWaitExceeded: {
          afterMs: 86_400_000,
          reaction: "notify-author",
        },
        tokenBudgetExceeded: {
          severity: "warning",
          reaction: { kind: "pause" },
        },
      },
    },
  },
);
```

The merged policy is stored on `workflow.opts.alertPolicy`.

## Merge Semantics

- `defaults` merge shallowly.
- `defaults.labels` merge by key.
- `rules` merge by rule name, and workflow-level fields win when the same rule appears in both places.
- `reactions` merge by reaction name, and workflow-level definitions replace module-level definitions with the same name.

## Reaction Kinds

| Kind | Use for |
| --- | --- |
| `emit-only` | Record the alert without pausing, cancelling, or external delivery. |
| `pause` | Pause the run for operator intervention. |
| `cancel` | Stop the run when continuing would be unsafe. |
| `open-approval` | Escalate into a human approval step. |
| `deliver` | Send the alert to a logical destination handled by your runtime integration. |

You can reference a named reaction from `rules` or inline a reaction object directly.

## Built-In Rules

The following rule names have built-in runtime evaluators. When configured in `alertPolicy.rules`, the engine automatically evaluates them against live events and persisted state.

| Rule Name | Fires When | Default Severity |
| --- | --- | --- |
| `runFailed` | Run status becomes `failed` | `critical` |
| `approvalWaitExceeded` | An approval has been pending longer than `afterMs` | `warning` |
| `retryStorm` | A node retries more than `threshold` times within `windowMs` | `warning` |
| `taskHeartbeatTimeout` | A task heartbeat timeout event is emitted | `warning` |
| `tokenBudgetExceeded` | A `BudgetExceeded` event fires for token, cost, or time | `critical` |
| `workflowReloadFailed` | Hot reload fails to apply | `warning` |
| `workflowReloadUnsafe` | Hot reload is skipped as unsafe | `info` |
| `providerDisconnected` | An agent provider disconnects without reconnecting | `warning` |

### Extended Rule Fields

Rules accept optional fields for threshold and escalation control:

```ts
rules: {
  retryStorm: {
    threshold: 10,         // fire after this many retries
    windowMs: 120_000,     // within this time window
    escalateAfterMs: 600_000, // escalate severity after this age
    escalateTo: "critical",   // target severity on escalation
    silenceForMs: 300_000,    // suppress delivery for this duration after silencing
  },
}
```

## Runtime Behavior

Alert policy rules are evaluated at runtime inside the workflow engine:

1. The effective merged `alertPolicy` is persisted in the run's `configJson` at start and resume, making evaluation deterministic and inspectable.
2. Built-in rules subscribe to the `EventBus` and poll time-based conditions (e.g., approval wait age) every 10 seconds.
3. Alerts are persisted to `_smithers_alerts` with fingerprint-based dedupe. Recurring identical conditions reopen resolved alerts instead of creating duplicates.
4. Reactions execute when an alert first fires: `pause` creates a linked human request, `cancel` aborts the run, `open-approval` creates an operator decision request.
5. All reactions are idempotent across resume and replay.

## Alert Lifecycle Events

The following events are emitted during the alert lifecycle and persisted like other `SmithersEvent` records:

- `AlertFired` — A new alert fires or an existing one reopens.
- `AlertAcknowledged` — An operator acknowledges an alert.
- `AlertSilenced` — An alert is silenced with an optional TTL.
- `AlertResolved` — An alert is resolved (manually or by system).
- `AlertReopened` — A previously resolved alert recurs.
- `AlertEscalated` — An alert's severity increases due to escalation rules.

## Unified Attention Model

Alerts, pending approvals, and alert-generated human requests are surfaced through a unified attention API:

- **Gateway**: `attention.list` and `attention.act` RPCs, plus real-time websocket events.
- **REST**: `GET /v1/attention` returns a combined list sorted by severity and age.
- **Web**: `/inbox` shows all attention items with filters, actions, and deep links.
- **CLI**: `smithers inbox` lists and acts on all attention items.
- **TUI**: The `AttentionPane` component shows live attention items with hotkeys.
- **PI**: Status bar shows total attention count; `ctx.ui.notify()` fires for critical alerts.

## Delivery and Suppression

- One active alert per fingerprint. Recurrence increments `occurrenceCount` and reopens.
- Silence expires automatically after `silencedUntilMs`.
- Focus suppression: toast, bell, and PI notifications are suppressed when the relevant run or inbox is already visible. Badge and inbox state always update.
- Severity escalation is time-based and configurable per rule.

## Recommended Usage

- Use alert policy for operational conditions such as failed runs, approval timeouts, or budget breaches.
- Keep destinations logical and stable, for example `oncall`, `author`, or `incident-feed`.
- Put transport details in your notifier boundary, not in workflow code.
- Avoid using alert policy for every ordinary business notification.
